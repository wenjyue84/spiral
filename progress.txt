## Pre-analysis for US-012 (Gemini 2.5 Pro)
Here is a technical brief for implementing story US-012.

**Objective:** Modify `ralph.sh` to append a telemetry row to a central `results.tsv` file after each story generation attempt.

**Affected Files & Hooks:**
*   **Primary Target:** `ralph/ralph.sh`. This script will be modified to contain the telemetry logging logic.
*   **No Frappe/Python:** This is a shell scripting task. No DocTypes, Python controllers, or Frappe hooks are involved. `spiral_dashboard.py` is a consumer of the generated file, not the producer.

**Implementation Approach:**
1.  **Locate Target:** In `ralph.sh`, find the main loop where the AI model is called and the subsequent acceptance test is run.
2.  **Time Capture:** Before the AI call, capture the start time: `STORY_START=$(date +%s)`.
3.  **Status & Duration:** After the test run, determine the `status`. A zero exit code from the test runner should map to 'keep', non-zero to 'reject'. Then calculate `DURATION=$(( $(date +%s) - STORY_START ))`.
4.  **File & Header:** Define the output file path, e.g., `RESULTS_TSV="$SPIRAL_HOME/results.tsv"`. Check if this file exists. If not, write the header row: `echo -e 'timestamp\t...' > "$RESULTS_TSV"`.
5.  **Append Data:** Use `printf` to append the tab-separated values. All variables (`story_id`, `model`, `retry_num`, `spiral_iter`) should be available as environment variables within the script's scope.

**Gotchas & Edge Cases:**
*   **Concurrency:** Workers run in parallel. While `>>` appends are generally atomic, using `flock` around the write block is safer to prevent interleaved lines from multiple `ralph.sh` instances.
*   **Pathing:** The path to `results.tsv` must resolve to the project root (`SPIRAL_HOME`), not the local worker directory.
*   **Environment:** Verify that `spiral.sh` correctly exports `SPIRAL_ITER` so it's available to the `ralph.sh` process.
